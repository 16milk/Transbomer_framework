# MiniTensor 开发计划

## 1. 开发策略

MiniTensor 采用“先闭环、再扩展”的开发方式。每个阶段都必须满足：

- 有明确的功能边界；
- 有单元测试或数值对照；
- 有可运行的示例；
- 不依赖尚未完成的后续模块；
- 通过验收后再进入下一阶段。

建议优先使用 Python 3.11+ 和 NumPy。测试框架使用 pytest，代码质量工具可选择
Ruff 和 mypy。第一版不引入 C++、CUDA 和分布式依赖。

## 2. 阶段总览

| 阶段 | 主要成果 | 建议周期 |
| --- | --- | --- |
| 0. 工程初始化 | 包结构、测试、CI、编码约定 | 2～3 天 |
| 1. Tensor 与基础算子 | 可执行的张量计算库 | 1～2 周 |
| 2. 自动微分 | 支持反向传播的动态计算图 | 2 周 |
| 3. 神经网络训练 | MLP 训练闭环 | 1～2 周 |
| 4. 模型组件完善 | CNN/Transformer 所需算子 | 2～3 周 |
| 5. 静态图与优化 | 图追踪、执行与基础优化 | 3～4 周 |
| 6. 序列化与工具 | 参数保存、加载和调试能力 | 1～2 周 |
| 7. RAG 综合示例 | 小型检索增强推理链路 | 2～3 周 |
| 8. 性能扩展 | C++、并行或 GPU 原型 | 按需开展 |

周期是单人业余开发的粗略估计，应以阶段验收结果为准。

## 3. 阶段 0：工程初始化

### 目标

建立可以持续开发和验证的项目骨架。

### 任务

- 建立 `src/minitensor` 包结构。
- 建立 `tests`、`examples` 和 `benchmarks` 目录。
- 配置 `pyproject.toml`、pytest 和代码检查工具。
- 定义代码风格、异常类型和公共 API 导出规则。
- 配置最小 CI：安装、静态检查、单元测试。
- 编写贡献指南和本地开发命令。

### 验收

- 全新环境可以安装项目。
- `import minitensor` 成功。
- CI 能执行一个最小测试。

## 4. 阶段 1：Tensor 与基础算子

### 目标

完成不带自动微分的张量计算层。

### 任务

- 实现 `Tensor` 的数据、shape、ndim、size 和 dtype。
- 明确数据所有权、复制和视图语义。
- 实现索引、切片、reshape、transpose 和广播。
- 实现 `add`、`sub`、`mul`、`div`、`pow`。
- 实现 `sum`、`mean`、`max` 等归约算子。
- 实现 `matmul`、`exp`、`log`、`relu`、`sigmoid`。
- 定义算子参数校验和错误信息规范。

### 测试重点

- 标量、向量、矩阵和高维 Tensor。
- 不同维度的广播。
- 非法 shape、axis 和 dtype。
- 与 NumPy 结果对齐。
- 浮点误差使用合理的容差比较。

### 验收

- 基础算子覆盖后续 MLP 所需的全部前向计算。
- 测试结果与 NumPy 一致。

## 5. 阶段 2：自动微分

### 目标

实现动态计算图和反向模式自动微分。

### 任务

- 为 Tensor 增加 `requires_grad`、`grad` 和计算来源。
- 定义 Operation/Context 抽象，保存反向计算所需信息。
- 构建计算图并通过拓扑排序执行反向传播。
- 实现梯度累积和 `zero_grad`。
- 实现广播、reshape、transpose、归约和 matmul 的梯度。
- 支持 `detach` 和禁用梯度记录的上下文。
- 处理共享子图和一个 Tensor 被多次使用的情况。
- 明确是否支持原地修改；第一版建议禁止影响计算图的原地操作。

### 测试重点

- 使用中心有限差分进行梯度检查。
- 测试分支图、共享节点和梯度累积。
- 测试标量输出约束和显式传入上游梯度。
- 测试广播维度的梯度归约。

### 验收

- 所有可微基础算子通过梯度检查。
- 复合表达式的梯度与参考实现一致。

## 6. 阶段 3：神经网络训练闭环

### 目标

让用户能够定义、训练和评估一个完整 MLP。

### 任务

- 实现 `Parameter` 和 `Module`。
- 实现参数自动注册和嵌套模块遍历。
- 实现 `Linear`、`ReLU`、`Sequential`。
- 实现 MSE 和交叉熵损失。
- 实现 SGD、Momentum 和 Adam。
- 实现 `train`/`eval` 模式。
- 提供线性回归、二分类和多分类示例。

### 验收

- 线性回归能够恢复合成数据参数。
- MLP 能在 XOR 或简单分类数据集上收敛。
- 损失下降、参数更新和梯度清零行为符合预期。

## 7. 阶段 4：模型组件完善

### 目标

补齐小型 Transformer 所需的数学算子和神经网络模块。

### 任务

- 实现稳定版本的 Softmax、LogSoftmax 和交叉熵。
- 实现 Embedding、LayerNorm、Dropout。
- 实现 batched matmul、mask 和必要的索引算子。
- 实现 Multi-Head Attention。
- 实现位置编码、前馈网络和 Transformer Block。
- 建立小型语言模型或序列分类示例。

### 测试重点

- 极大或极小输入下的数值稳定性。
- train/eval 模式下 Dropout 的差异。
- attention mask 的 shape 和广播行为。
- LayerNorm 和 Attention 与可信参考结果对齐。

### 验收

- 小型 Transformer 能完成前向和反向传播。
- 在小数据集上可以过拟合，以证明训练链路正确。

## 8. 阶段 5：静态图与图优化

### 目标

在即时执行之外，提供可分析和优化的静态计算图。

### 任务

- 定义图 IR：Value、Node、Graph 和属性。
- 设计基于示例输入的追踪或显式建图机制。
- 实现 shape/dtype inference。
- 实现拓扑排序和静态图执行器。
- 实现常量折叠和死代码消除。
- 选择实现公共子表达式消除或简单算子融合。
- 比较即时执行和静态图执行结果。
- 输出文本化图结构，便于调试。

### 验收

- 同一模型在即时模式和图模式下结果一致。
- 图优化前后的结果一致。
- 测试能够证明优化减少了节点数或执行开销。

## 9. 阶段 6：序列化与开发工具

### 目标

提高模型复用能力和框架可调试性。

### 任务

- 保存和加载 `state_dict`。
- 记录模型结构、参数 shape 和 dtype。
- 为不兼容参数提供明确错误信息。
- 提供随机种子控制。
- 提供参数量、计算图节点和简单耗时统计。
- 建立基准测试脚本，对比 NumPy 或参考框架。

### 验收

- 训练后的模型保存并加载后输出一致。
- 固定随机种子后测试和示例可重复。

## 10. 阶段 7：RAG 综合示例

### 目标

用一个边界清晰的小型 RAG 链路检验框架的综合能力。

### 推荐范围

- 使用少量本地 Markdown 或纯文本文档。
- 实现固定长度或按段落切分。
- 使用预计算向量，或用 MiniTensor 的小型编码器生成向量。
- 使用余弦相似度和 NumPy 完成内存检索。
- 使用小型 Transformer 完成分类、重排或受限文本生成。
- 输出查询、命中文档、相似度和模型结果。

### 不在此阶段实现

- 生产级向量数据库；
- 网络爬虫和复杂文档解析；
- 大模型训练；
- 高并发服务和多租户管理；
- 对外部大模型供应商的深度封装。

### 验收

- 示例能够通过一条命令运行。
- 固定输入和随机种子时结果可重复。
- 能展示 MiniTensor 在编码、重排或生成环节的实际作用。

## 11. 阶段 8：性能与后端扩展

该阶段只在核心功能稳定后开展。可以选择一个方向深入：

- 使用 pybind11 实现 C++ CPU Kernel；
- 对矩阵计算进行多线程或 SIMD 优化；
- 设计统一 Device 和 Backend 接口；
- 实现少量 CUDA Kernel；
- 研究算子融合和内存复用；
- 增加性能分析器。

不要同时启动全部方向。应先通过 benchmark 定位主要瓶颈，再选择收益明确的优化。

## 13. 质量要求

### 正确性

- 每个算子必须有前向测试。
- 每个可微算子必须有梯度检查。
- 关键模型组件应与 NumPy 或可信框架结果对照。

### 可维护性

- Tensor、自动微分、神经网络和后端之间保持单向依赖。
- 公共 API 与内部实现分离。
- 新增算子时应同时定义前向、反向、shape 规则和测试。

### 性能

- 第一版不以绝对性能作为验收条件。
- 所有性能优化必须有 benchmark 证明收益。
- 不用牺牲梯度正确性或 API 清晰度换取未经测量的优化。

## 14. 主要风险

| 风险 | 应对方式 |
| --- | --- |
| 项目范围不断扩大 | 按阶段冻结需求，扩展能力进入后续里程碑 |
| 梯度静默出错 | 强制有限差分检查和参考实现对照 |
| API 过早固化 | 前三阶段将公共 API 保持精简 |
| NumPy 细节掩盖框架机制 | 明确 NumPy 只作为初始计算后端 |
| Transformer 阶段复杂度过高 | 先完成 MLP 闭环，再逐个增加组件 |
| RAG 偏离框架主线 | 将其限定为验收示例，不自研外围基础设施 |
| 性能优化投入过早 | 核心正确性稳定后再使用 benchmark 决策 |

## 15. 第一版本发布条件

建议将完成阶段 0～3 定义为 `v0.1.0`：

- 可以安装和导入；
- Tensor 基础运算正确；
- 自动微分通过梯度检查；
- 支持 Module、Linear、常用损失和优化器；
- 至少一个 MLP 示例能够稳定收敛；
- CI 覆盖测试与代码检查；
- 文档说明已实现能力和已知限制。
